## ADR-001 · Публичный моно-репо + правило санитарии
**Дата:** 2026-07-10 · **Статус:** accepted
**Контекст:** raw-URL без токенов — нулевая настройка для сессий (проверено на dom-prorab-catalog); но проект содержит контур клиента.
**Решение:** репо `lombard-ops` публичный, docs+код вместе. ЗАПРЕЩЕНО в репо: креды/ключи любого рода (в т.ч. VPN-конфиги WireGuard), публичные endpoint'ы контура клиента (DDNS-имена, внешние порты VPN), токены Telegram, ПДн клиентов ломбарда (имена/телефоны/номера договоров в примерах — обезличивать). ДОПУСТИМО: внутренние LAN-IP, номера портов сервисов, PROJECT_ID, имена ресурсов GCP.
**Последствия:** дампы структуры БД перед коммитом санитизируются; секреты — Secret Manager; примеры данных — синтетика.

## ADR-002 · Единое Cloud Run приложение + Identity Platform вместо Apps Script / GCS static / cf-api
**Дата:** 2026-07-10 · **Статус:** accepted
**Контекст:** требование логин/пароль + роли несовместимо с Apps Script «Anyone»; драфт ТЗ размазывал UI по трём технологиям и сам помечал Apps Script на снос при росте; signed-URL через HMAC в Apps Script — самый хрупкий пункт драфта.
**Решение:** один Cloud Run сервис `app-lombard` (React + API); auth — Identity Platform (email+пароль), роли `owner`/`manager`/`assessor` в custom claims; API проверяет токен и роль на каждом запросе. Все экраны — роуты одного приложения.
**Последствия:** cf-api, GCS static hosting и Apps Script не строятся; фото — обычные signed URL из бэкенда.

## ADR-003 · Ingestion: pull-коннектор из БД Firebird; fallback — агент на сервере
**Дата:** 2026-07-10 · **Статус:** accepted (канал доставки — pending, см. ADR по T-0-8)
**Контекст:** дано прямое read-only подключение к БД PawnShop; file-drop драфта делал Бектура ручным звеном и порождал staleness-проблему.
**Решение:** данные забирает `job-connector` (Cloud Run Job) по расписанию напрямую из Firebird. Ручная загрузка файлов не строится. Запасные каналы: (B) агент на ERP-сервере с только исходящим пушем в GCS; (C) штатная выгрузка PawnShop «в бух. программы».
**Последствия:** дискавери смещается на схему БД и сеть; staleness-проверка остаётся как guard обрыва связи.

## ADR-004 · Guard схемы Firebird
**Дата:** 2026-07-10 · **Статус:** accepted
**Контекст:** вендор меняет структуру БД при обновлениях и не уведомляет.
**Решение:** перед каждым чтением — отпечаток метаданных используемых таблиц (RDB$) сверяется с эталоном в mapping.json; расхождение → стоп + алерт, чтения нет. `SELECT *` запрещён. Обновление эталона — осознанная правка конфига.
**Последствия:** обновление PawnShop ломает конвейер громко и безопасно, а не тихо портит данные.

## ADR-005 · Порядок фаз: конвейер данных раньше форм
**Дата:** 2026-07-10 · **Статус:** accepted
**Контекст:** драфт ставил форму осмотра первой (логично при file-drop); при pull-коннекторе данные доступны сразу, а форма требует готового приложения с auth. Урок референс-проекта: интегральный путь проверять рано.
**Решение:** Фаза 1 — конвейер + светофор с tracer bullet в конце фазы; Фаза 2 — приложение и формы; Фаза 3 — Stop-Loss.
**Последствия:** Иса получает ценность (сводка, алерты) раньше; допущения о данных проверяются первыми.

## ADR-006 · balance_amount вычисляется на нашей стороне
**Дата:** 2026-07-10 · **Статус:** accepted
**Контекст:** ответ вендора: начисления (%/пени) в БД не хранятся, считает приложение.
**Решение:** балансовая стоимость вычисляется в cf-daily из условий договора (тело, ставка, пени-правила — маппинг T-0-3) по формулам `03 §4`. Валидация: сверка на нескольких договорах с цифрами из UI PawnShop.
**Последствия:** дискавери обязан найти поля условий; расхождение с PawnShop UI — блокер запуска Фазы 3.

## ADR-007 · Наследование конвенций dom-prorab-catalog
**Дата:** 2026-07-10 · **Статус:** accepted
**Контекст:** дисциплина (anti-improvisation, session-блоки, модель исполнения, untrusted-вывод, не-идемпотентные операции) доказана референс-проектом.
**Решение:** `05_CONVENTIONS` наследует правила референса; отличия фиксируются только явно в `05`/ADR. `10_GCP_INFRA_PLAYBOOK` пополняется переносом релевантных разделов из референса по мере надобности.
**Последствия:** новые грабли GCP сначала ищутся в плейбуке референса, потом решаются заново.

## ADR-008 · Техдолг сети клиента (фиксация, без действия сейчас)
**Дата:** 2026-07-10 · **Статус:** accepted
**Контекст:** WG-подсеть `172.168.88.0/24` — публичный диапазон (админ, вероятно, целил в приватный 172.16–31); VPN-пароль SSTP-пользователя засвечен в переписке; ERP-сервер без ИБП.
**Решение:** зафиксировать как техдолг; закрыть на T-4-4 (смена пароля, предложение админу пересадить WG на приватный диапазон, рекомендация Исе по ИБП). Работу сейчас не блокирует.
**Последствия:** чек-лист гигиены в Фазе 4.

## ADR-009 · Дампы структуры БД вендора не коммитятся в публичный репо
**Дата:** 2026-07-14 · **Статус:** accepted
**Контекст:** вендор «Алгоритм» считает схему БД конфиденциальной; полный дамп структуры (108 таблиц, 1018 колонок) получен при выполнении T-0-2.
**Решение:** дампы структуры БД передаются архитектору только в чат (вне репо). В репо попадает исключительно маппинг используемых полей (документация `02 §3`). Синтетические примеры в кодексе/конфигах — без специфичных колонок вендора.
**Последствия:** T-0-3 и далее работают с производным маппингом (`mapping.json`), не с полным дампом; дамп остаётся в истории чата как справка для аналитики, недоступный для public github.
