**Версия:** 1.0 · **Статус:** SEMI-STABLE · **обновлено:** 2026-07-10
**Самый важный документ для переносимости. Читается первым в любой сессии.** Наследует конвенции проекта dom-prorab-catalog (ADR-007); отличия — только зафиксированные здесь.

## ЧАСТЬ I — ПРАВИЛА АГЕНТА

### Роли
- **Opus** — архитектор. Проектирует, планирует, правит документацию, принимает решения (ADR). Не пишет продакшен-код задач.
- **Sonnet** — разработчик. Берёт ОДИН task-brief, пишет код/SQL/скрипты. **Не исполняет** — исполняет человек.
- **Человек (Ilyas)** — исполнитель и владелец решений. Запускает код (Cloud Shell / терминал macOS / RDP на ERP-сервере), возвращает логи, коммитит патчи в репо.
- **Applier (Haiku, опц.)** — механически применяет session-блоки к артефактам репо.

### Модель исполнения (железно)
Sonnet **пишет** код/SQL/скрипты с ожидаемым выводом и инструкцией запуска. Человек **запускает** и возвращает логи. Sonnet анализирует логи → следующий шаг или починка. Sonnet НИКОГДА не считает задачу выполненной без подтверждённого человеком лога.

### Ритуал СТАРТА сессии
1. Загрузить обязательный контекст: `00 + 05 + 07 + 11` + конкретный task-brief + всё из его «context-to-load».
2. Переформулировать задачу своими словами; перечислить полученные входы.
3. Проверить полноту; чего-то нет → anti-improvisation.
4. Бриф впервые поднимает тип ресурса с разделом в `10_GCP_INFRA_PLAYBOOK` → раздел идёт в context-to-load.

### ★ Anti-improvisation (главный закон проекта)
Требуемый документ, вход, значение или решение отсутствует в контексте → модель **ОСТАНАВЛИВАЕТСЯ**:
```
CONTEXT GAP: <что отсутствует и почему без него нельзя продолжить>
```
Запрещено выдумывать, «дотягивать», подставлять правдоподобные значения, угадывать имена таблиц/пороги/схемы. Особо для этого проекта: **схему БД PawnShop не угадывать никогда** — только факты из дампа структуры или живого запроса.

### ★ Untrusted-вывод терминала
Внутренне противоречивый/повреждённый вывод (paste-артефакты RDP и Cloud Shell, рассинхрон тела и статуса) — НЕДОСТОВЕРЕН. Не примирять противоречие и не выбирать «правдоподобную половину» — запросить чистый повторный лог в разделимой форме.

### ★ Не-идемпотентные операции
Обрыв/timeout во время `create`, `deploy`, `jobs run` → слепой retry запрещён. Сначала read-only диагностика (describe/list), затем доисполнение только недостающего хвоста.

### ★ Только чтение из PawnShop
Любой SQL к Firebird — только SELECT (+ чтение системных RDB$-таблиц). Sonnet не выдаёт человеку для запуска на БД PawnShop ничего кроме чтения; единственное исключение — создание read-only пользователя и GRANT SELECT (T-0-5) по явному брифу.

### Ритуал КОНЦА сессии
Завершить структурированным session-блоком (Часть III). Без блока сессия не закрыта, знание потеряно.

### Дисциплина документации
- `00, 05, 08, 09` — правки только через ADR. `01–03` — SEMI-STABLE, правки через ADR или session-блок с пометкой.
- `06` — только append; новое решение = новый ADR, старый помечается superseded.
- `07` — каждую сессию через STATE_PATCH. `11` — через INFRA_PATCH (даже пустой, явным «нет изменений»).
- `04` — по мере исполнения через session-блок.

## ЧАСТЬ II — ТЕХНИЧЕСКИЕ СТАНДАРТЫ

### Структура репозитория
```
/            00–11 .md — база знаний (плоско)
/briefs      _GENERATOR.md + брифы задач
/sql         ddl/*.sql, status.sql, pricing.sql, dq.sql
/connector   Firebird → BigQuery (Cloud Run Job)
/functions   /cf-daily, /cf-tg-handler — main.py, requirements.txt
/app         Cloud Run приложение (React + API)
/scripts     setup gcloud-блоки, сиды, одноразовые
```

### Нейминг
- GCP: проект `project-c451b48a-07ae-4de4-961` (факт, см. `11`); бакеты `${PROJECT_ID}-{photos|config|cfsource}`; функции `cf-<verb>`; Run job `job-connector`; Run service `app-lombard`; scheduler `sch-<name>`; SA `lombard-pipeline@`.
- BQ: датасет `lombard_ops`; таблицы snake_case.
- GCS-фото: `photos/{VIN}/{timestamp}/`; тест-фикстуры — только префикс `test-fixtures/`, никогда не в боевых префиксах.
- Идентификаторы/колонки — английский snake_case; проза для людей — русский.

### Код-стандарт
- SQL: standard BigQuery SQL; каждое правило — комментарий-ссылка на `03 §`; пороги из конфига, не хардкод.
- Python: минимум зависимостей; gate'ы блокирующие (raise), не warning; явные схемы при load в BQ (SchemaField, не autodetect).
- Запросы к Firebird: явные списки колонок, `SELECT *` запрещён; перед чтением — guard схемы (ADR-004).
- Shell — самодостаточный copy-paste (heredoc); GUI-шаг, где неизбежен, — отдельным явным шагом с точной навигацией.
- Деплой cf-*/Run — только из чистой выделенной папки, не `--source=.` из каталога с посторонним.
- Секреты — только Secret Manager. В коде и репо — никогда.

### Доступ к документации в чате
- Чтение: raw-URL `https://raw.githubusercontent.com/ilyasbazarov/lombard-ops/main/<FILE>.md`.
- Перечитывание внутри сессии — пиннить по commit-SHA (обходит CDN-кэш main). SHA недоступен → запросить прямую вставку текста.
- Запись — НЕ прямым коммитом от модели. Только session-блок → человек/applier коммитит.

### Коммиты
`<область>: <что> (T-<id>)` · документы: `docs: state patch <date>` / `docs: ADR-0XX <тема>`.

## ЧАСТЬ III — ФОРМАТЫ SESSION-БЛОКОВ

В конце каждой сессии — ОДИН блок. Applier/человек применяет механически.

```md
=== SESSION LOG · <date> · <task-id> ===

## SESSION_LOG
- Задача: <T-id> — <название>
- Сделано: <маркеры>
- Команды/логи ключевые: <кратко>
- Отклонения от плана: <если были, иначе "нет">

## STATE_PATCH  (→ 07_STATE.md)
- Задача <T-id>: <старый статус> → <новый статус>
- Текущий фокус: <следующая задача>
- Новые открытые вопросы: <список или "нет">
- Блокеры: <список или "нет">
- updated_at: <date>

## INFRA_PATCH  (→ 11_INFRA_FACTS.md; обязателен всегда, пустой = явное "нет изменений")
- <ресурс: значение/статус или "нет изменений">

## NEW_DECISIONS  (→ append в 06 как ADR; "нет" если нет)
- ADR-0XX: <заголовок> · Контекст · Решение · Последствия · Статус: accepted | proposed

## NEW_CONVENTIONS  (→ предложение правки 05; "нет" если нет; proposed — не применять без апрува)

=== END SESSION ===
```

`STATE_PATCH` и `INFRA_PATCH` применяются всегда. `NEW_DECISIONS`/`NEW_CONVENTIONS` со статусом proposed ждут апрува Ilyas или Opus-сессии.

## ЧАСТЬ IV — ГЕНЕРАЦИЯ БРИФОВ

Ни один чат не держит весь проект. «Архитектор» — роль, не чат: свежий Opus, загрузивший `00+04+06+07`, становится архитектором; старые чаты ретайрятся.

- **Механический бриф** (задача расписана в `04`) → свежий Sonnet через `briefs/_GENERATOR.md`.
- **Бриф с суждением** (нужны решения) → свежий Opus: сначала закрывает открытые пункты (ADR), потом генерит.

Процедура генератора: `07` → фокус → `04` → зависимости и приёмка → `08` + context-to-load → самодостаточный бриф (raw-URL по SHA, приёмка, return-this, session-блок).
